# Чеклист качества промпта

## Описание

Чеклист для составления эффективных промптов к AI-инструментам (MOEX GPT, Insight, Kilo Code). Помогает получить более точные, полные и полезные ответы от AI. Применим для любых задач -- от написания кода до подготовки аналитических отчётов.

---

## 1. Контекст задачи

- [ ] **Роль определена** -- указано, в какой роли должен действовать AI
  - Пример: "Ты -- senior Python-разработчик с опытом в финансовых системах"
- [ ] **Контекст проекта описан** -- AI понимает, для чего нужен результат
  - Пример: "Это внутренний портал MOEX AI Hub для обучения сотрудников работе с AI"
- [ ] **Целевая аудитория указана** -- для кого предназначен результат
  - Пример: "Документ для менеджеров среднего звена без технического бэкграунда"

## 2. Чёткость запроса

- [ ] **Задача конкретна** -- нет расплывчатых формулировок
  - Плохо: "Напиши что-нибудь про API"
  - Хорошо: "Создай FastAPI эндпоинт для получения списка пользователей с пагинацией"
- [ ] **Объём определён** -- указан ожидаемый размер результата
  - Пример: "Ответ на 5-7 пунктов", "Функция не более 30 строк"
- [ ] **Формат указан** -- описан желаемый формат вывода
  - Пример: "В формате Markdown-таблицы", "В виде Python-кода с docstrings"

## 3. Ограничения и требования

- [ ] **Технологический стек указан** -- версии, фреймворки, библиотеки
  - Пример: "Python 3.11, FastAPI, SQLAlchemy 2.0, PostgreSQL 15"
- [ ] **Стандарты определены** -- coding style, naming conventions
  - Пример: "Следуй PEP 8, используй type hints, Google-style docstrings"
- [ ] **Ограничения явные** -- что НЕ нужно делать
  - Пример: "Не используй ORM, только raw SQL через asyncpg"
- [ ] **Безопасность учтена** -- нет запроса на генерацию секретов или небезопасного кода

## 4. Примеры (Few-shot)

- [ ] **Входные примеры предоставлены** -- AI видит ожидаемый формат
  - Пример: "Вот пример существующего эндпоинта, создай аналогичный для /trades"
- [ ] **Примеры хорошего результата** -- AI понимает стандарт качества
- [ ] **Примеры плохого результата** -- AI знает, чего избегать

## 5. Декомпозиция

- [ ] **Сложная задача разбита на шаги** -- не пытаться решить всё одним промптом
  - Шаг 1: "Спроектируй схему данных для модуля квизов"
  - Шаг 2: "Создай SQLAlchemy модели по этой схеме"
  - Шаг 3: "Напиши API эндпоинты для CRUD операций"
- [ ] **Промежуточные результаты проверяются** -- валидация на каждом шаге

## 6. Итерация и уточнение

- [ ] **Первый результат оценён критически** -- не принимать первый ответ как финальный
- [ ] **Feedback конкретен** -- "Добавь обработку ошибок" вместо "Сделай лучше"
- [ ] **Рефинемент направлен** -- указать конкретно, что изменить
  - Пример: "Добавь валидацию email через regex и ограничение длины username до 50 символов"

## 7. Верификация результата

- [ ] **Факты проверены** -- ссылки, названия API, синтаксис команд
- [ ] **Код протестирован** -- запустить, не просто прочитать
- [ ] **Нет "галлюцинаций"** -- несуществующих функций, библиотек, API
- [ ] **Результат соответствует стандартам проекта** -- code style, архитектура

## Шаблон хорошего промпта

```
РОЛЬ: [Кем должен выступить AI]

КОНТЕКСТ: [Описание проекта и задачи]

ЗАДАЧА: [Конкретное действие]

ТРЕБОВАНИЯ:
- [Технологии / стек]
- [Стандарты и стиль]
- [Ограничения]

ФОРМАТ ОТВЕТА: [Как оформить результат]

ПРИМЕРЫ: [Входы и ожидаемые выходы, если применимо]
```

## Антипаттерны промптов

| Антипаттерн | Проблема | Решение |
|------------|----------|---------|
| "Напиши код" | Слишком расплывчато | Указать язык, задачу, контекст |
| "Сделай красиво" | Субъективно | Указать конкретные критерии |
| "Исправь баг" | Нет контекста | Приложить код, ошибку, ожидаемое поведение |
| Весь код в одном промпте | Перегрузка | Разбить на шаги |
| Без примеров | AI угадывает формат | Дать 1-2 примера |
