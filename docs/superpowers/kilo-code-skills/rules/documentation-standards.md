# Documentation Standards -- MOEX AI Hub

## Описание

Стандарты документации для проекта MOEX AI Hub. Определяют требования к документированию кода, API, архитектуры и процессов. Хорошая документация снижает порог входа для новых разработчиков и уменьшает количество вопросов в чатах.

## Уровни документации

### 1. Код (inline)

**Docstrings** обязательны для:
- Всех публичных функций и методов
- Всех классов
- Всех модулей (верхний уровень файла)

**Формат: Google-style docstrings**

```python
def calculate_portfolio_risk(
    positions: list[Position],
    market_data: MarketData,
    confidence_level: float = 0.95,
) -> RiskMetrics:
    """Рассчитывает метрики риска портфеля методом VaR.

    Использует историческую симуляцию для расчёта Value at Risk
    и Expected Shortfall на основе текущих позиций и рыночных данных.

    Args:
        positions: Список открытых позиций портфеля.
        market_data: Актуальные рыночные данные для расчёта.
        confidence_level: Уровень доверия для VaR (по умолчанию 95%).

    Returns:
        RiskMetrics с полями var, es, sharpe_ratio.

    Raises:
        InsufficientDataError: Если исторических данных менее 252 дней.
        InvalidPositionError: Если позиция содержит некорректный ISIN.
    """
```

**Комментарии в коде:**
- Объясняют ЗАЧЕМ, а не ЧТО
- Обязательны для нетривиальной бизнес-логики
- Обязательны для workaround-решений (с ссылкой на issue)


### 2. API (автогенерация + дополнения)

**FastAPI OpenAPI** -- автогенерируемая документация, обогащённая вручную:

- `summary` -- краткое описание (для списка эндпоинтов)
- `description` -- подробное описание (для детальной страницы)
- `tags` -- группировка по доменам
- `response_model` -- схема ответа
- `responses` -- описание нестандартных ответов (ошибки)

```python
@router.post(
    "/submissions",
    response_model=SubmissionResponse,
    status_code=201,
    summary="Создать заявку на контент",
    description="Позволяет пользователю подать заявку на новый обучающий материал. "
                "Заявка проходит модерацию перед публикацией.",
    tags=["submissions"],
    responses={
        409: {"description": "Заявка с таким названием уже существует"},
        422: {"description": "Невалидные данные заявки"},
    },
)
```

### 3. Архитектура (docs/)

**Обязательные документы:**

| Документ | Описание | Обновление |
|----------|----------|------------|
| `TARGET_ARCHITECTURE.md` | Целевая архитектура системы | При изменении архитектуры |
| `ADR/` | Architecture Decision Records | При принятии решений |
| `API.md` | Обзор API и примеры использования | При изменении API |
| `DEPLOYMENT.md` | Инструкции по развёртыванию | При изменении процесса |

**Architecture Decision Records (ADR):**

```markdown
# ADR-001: Выбор PostgreSQL для хранения данных

## Статус
Принято (2026-02-15)

## Контекст
Необходимо выбрать СУБД для хранения пользовательских данных и контента.

## Решение
Используем PostgreSQL 15+ как основную СУБД.

## Обоснование
- ACID-совместимость для финансовых данных
- JSONB для гибких структур контента
- Зрелая экосистема и поддержка в MOEX

## Последствия
- Необходим DBA для оптимизации в продакшене
- Миграции через Alembic
```

### 4. Процессы (внутренние)

- README.md в корне проекта: quick start, prerequisites, как запустить
- CONTRIBUTING.md: правила работы с git, code review, стандарты
- CHANGELOG.md: история изменений (Keep a Changelog формат)

## Правила для AI-агента

1. При создании новой функции / класса -- ВСЕГДА добавлять docstring
2. При создании нового API-эндпоинта -- ВСЕГДА добавлять summary, description, tags
3. При изменении архитектуры -- ОБЯЗАТЕЛЬНО обновить документацию
4. При добавлении конфигурации -- ОБЯЗАТЕЛЬНО задокументировать в .env.example
5. Не генерировать placeholder-документацию ("TODO: add description")

## Язык документации

- Код: комментарии и docstrings на английском языке
- API описания: на русском или английском (единообразно в рамках проекта)
- Архитектурные документы: на русском языке
- README и CONTRIBUTING: на русском языке с английскими терминами

## Чеклист качества документации

- [ ] Docstrings актуальны (соответствуют текущей реализации)
- [ ] API описания содержат примеры запросов/ответов
- [ ] Нет "TODO: add docs" в production коде
- [ ] Архитектурные решения задокументированы в ADR
- [ ] .env.example содержит все необходимые переменные с описаниями
- [ ] README позволяет новому разработчику запустить проект за 15 минут
